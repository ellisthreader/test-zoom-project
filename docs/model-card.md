# Model Card — RelayClarity Case-Risk Classifier

## Overview

- **Name:** `risk-scoring-logreg`
- **Version:** `risk-scoring-logreg-1.0.0`
- **Type:** Multinomial logistic regression (scikit-learn).
- **Task:** Classify a support case / call as `low`, `medium`, or `high` risk to
  prioritise human review.
- **Status:** Advisory. The model never auto-closes, auto-routes, or makes a
  final decision. A human owner reviews and can override every recommendation.

## Intended use

- Triage a queue of tickets/calls so urgent or sensitive cases surface first.
- Flag cases for mandatory human review (high risk, low confidence, or
  explicit escalation).

## Out of scope

- Automated decisioning without human oversight.
- Use on real customer or resident data without an information-governance review
  (the published model is trained on **synthetic** data only).

## Data

- Fully synthetic, generated by `ml/generate_data.py`.
- Labels come from a noisy latent severity rule, so the model has a real signal
  to recover but is not a perfect oracle.
- Structured features: priority, customer tier, escalation flag, sentiment,
  category sensitivity, previous open tickets, AI confidence, call outcome,
  duration, channel. Text feature: counts of high/medium-risk wording.
- Feature contract: `ml/features.py` (mirrored in `server/ai/risk-features.tsx`).

## Performance (synthetic hold-out, 1,200 cases)

Source: `ml/artifacts/evaluation.json`.

| Metric | Value |
| --- | --- |
| Accuracy | 0.8042 |
| Macro F1 | 0.7692 |
| High-risk recall | 0.8541 |
| High-risk precision | 0.9206 |

High-risk recall is the headline metric: missing a genuinely high-risk case is
the costliest error, so it is gated and monitored.

**Deployment gate (must pass to promote):** accuracy ≥ 0.70, high-risk
recall ≥ 0.70, macro-F1 ≥ 0.65. Current model: **PASS**.

### Calibration

| Metric | Value |
| --- | --- |
| Method | Post-hoc temperature scaling |
| Temperature | 1.0446 |
| Expected Calibration Error (ECE) | 0.0167 |

The temperature is fitted on a held-out calibration slice and stored in
`risk_model.json`. It is applied **identically** in `ml/infer.py` and
`server/ai/risk-model.tsx`, so the confidence shown to officers is trustworthy —
which matters because the 0.6 low-confidence threshold drives mandatory human
review. Temperature scaling does not change the predicted class, so the accuracy
and recall above are unaffected.

## Fairness

Reported as the gap in high-risk rate across slices (`ml/evaluate.py`,
`ml/artifacts/evaluation.json`):

- Customer tier: **0.2059** — VIP cases (0.6793) are flagged high more often than
  standard (0.4734), driven by the legitimate tier feature; monitored, not
  auto-corrected. At the live warn line (0.20).
- Channel: **0.0691**.
- Category: **0.3252** — expected, since category sensitivity is a real risk
  driver.

These gaps are surfaced for human governance review rather than silently
optimised away, and are also tracked **live** on real traffic by
`customerTier`/`channel` in `server/ai/monitoring.tsx`
(`GET /api/monitoring/drift-report`), with warn ≥ 0.20 / alert ≥ 0.30.

## Explainability

Each prediction returns class probabilities, the top per-feature contributions
to the high-risk logit (SHAP-style), plain-English reasons, and the matched
risk terms. See `featureAttributions` and `mainReasons` in the API response.

## Monitoring

- Drift: population stability index of the live class mix vs the training
  baseline (warn 0.10 / alert 0.20) — `GET /api/monitoring/drift-report`.
- Fairness: live `customerTier`/`channel` high-risk-rate disparity (warn 0.20 /
  alert 0.30) and operational health, in the same drift report.
- Human feedback / override rate — `GET /api/monitoring/feedback-report`.
- Provenance — `GET /api/risk/model`.

## Serving & no train-serve skew

The same exported artifact (`ml/artifacts/risk_model.json`) is served three ways:
the in-app TypeScript scorer (`server/ai/risk-model.tsx`), the FastAPI service
(`serving/`, deployable to AKS), and the Azure ML managed online endpoint
(`azure/score.py`). A parity test (`server/ai/parity.test.tsx`) asserts the
TypeScript scorer reproduces the Python model on fixtures generated by
`ml/export_parity_fixtures.py`, proving there is no train-serve skew.

## Retraining

`ml/retrain_from_feedback.py` closes the field-to-model loop: it merges
officer-confirmed feedback with the synthetic set, retrains a candidate, and
promotes it **only** if all gates hold (no high-risk recall regression > 0.02,
no tier fairness regression > 0.02, plus the deployment gate). It is dry-run by
default and backs up the live artifact to `risk_model.prev.json` before any
promotion.

## Limitations

- Synthetic data does not prove real-world performance.
- Linear model cannot capture complex feature interactions.
- Keyword counts are shallow vs. full NLP; deliberate, for transparency.

## Owner & review

Rebuild and re-evaluate via `npm run ml:pipeline`. Review metrics, calibration,
fairness, and drift before promoting a new version. Roll back by restoring
`ml/artifacts/risk_model.prev.json` over `risk_model.json` (or shifting Azure ML
endpoint traffic to the previous deployment — see `azure/README.md` and
`docs/incident-runbook.md`). Full governance: `docs/responsible-ai.md` and the
full DPIA in `docs/dpia.md`.
